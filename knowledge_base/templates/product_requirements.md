# Product Requirements Document

## Purpose

A Product Requirements Document (PRD) defines the goals, features, constraints, and success criteria of a product before development begins.

## Recommended Structure

- Product Vision
- Problem Statement
- User Personas
- Functional Requirements
- Non-functional Requirements
- User Stories
- Acceptance Criteria
- KPIs

## Best Practices

- Focus on user needs.
- Define measurable goals.
- Keep requirements testable.
- Prioritize features.
- Avoid implementation details.

## Common Mistakes

- Mixing technical design with requirements
- Missing acceptance criteria
- Unclear feature priorities
- Undefined success metrics

## Example

User Story

"As a project manager, I want AI to generate project proposals so that documentation time is reduced."

## Summary

A PRD should clearly describe what needs to be built and why it provides value.
